# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

Package manager is **pnpm** (a `pnpm-lock.yaml` is committed). Node 24+.

| Command | Purpose |
| ------- | ------- |
| `pnpm dev` | Vite dev server (http://localhost:5173). |
| `pnpm build` | `tsc -b` then `vite build`. Type errors fail the build. |
| `pnpm typecheck` | `tsc --noEmit` — fastest correctness check; run after edits. |
| `pnpm lint` | ESLint. |
| `pnpm format` | Prettier (with Tailwind class sorting). |
| `pnpm preview` | Serve the production build. |

There is **no test runner** configured — do not assume `pnpm test` exists. Verify changes with `pnpm typecheck` and by running the app.

Database migrations are applied separately via the Supabase CLI (`npx supabase db push`) — see `supabase/README.md`.

## Conventions

- **Descriptive names, never single-letter.** Name every variable for what it holds — including loop, callback, and map/filter parameters. Write `leads.map((lead) => …)` and `prev.map((lead) => …)`, not `leads.map((l) => …)`. This applies to `.map/.filter/.reduce` callbacks and destructured shorthands too.
- **Always use the design tokens; never invent colors.** Style with the theme tokens from `src/index.css` (via Tailwind classes like `bg-card`, `text-muted-foreground`, `ring-primary`, `border-border`, or `var(--…)`). Do not hardcode hex/RGB/OKLCH values or reach for arbitrary Tailwind colors (`bg-[#10b981]`, `text-emerald-500`). If a design needs a color the tokens don't cover, **ask the user whether a new token should be added** to `src/index.css` (light + dark) rather than hardcoding one.
- **Tabular data → always `DataTable` with resizing.** Render any table with the shared `DataTable` (`src/components/data-table/`, exported from `@/components/data-table`) — never a hand-rolled `<table>`. Always pass the `resizable` prop so columns can be resized (see the Leads table in `src/routes/_authenticated/leads/index.tsx`).
- **Prefer reuse over new code.** Before adding a component, hook, utility, or type, check whether one already exists and use/extend it:
  - UI primitives → `src/components/ui/` (shadcn/ui: Card, Button, Badge, ScrollArea, DataTable, …). Feature components → `src/components/<feature>/`.
  - Class merging → `cn()` in `src/lib/utils.ts`. Domain formatting/metadata → `src/lib/` (e.g. `lead-format.ts`).
  - Data access → the existing `*.service.ts` (below). Types are re-exported from `@/services`.
  Only create something new when nothing fits, and match the surrounding idiom when you do.

## Architecture

React 19 + TypeScript SPA, Vite 8, backed by Supabase (Postgres + Auth). Import alias `@/*` → `src/*`.

### Routing (TanStack Router, file-based)
- Routes live in `src/routes/`; each exports a `Route` via `createFileRoute`. `src/routeTree.gen.ts` is **auto-generated by the router plugin — never edit it** (regenerates on dev/build).
- `_authenticated.tsx` is the auth guard: it redirects to `/login` when there's no session, hydrates the profile into the auth store, and wraps every child page in `MainLayout`. All app pages are folders under `src/routes/_authenticated/`.
- Adding a page = new folder under `_authenticated/` **plus** a nav entry in `src/components/app-sidebar.tsx` (nav items are a plain data array there).

### Data layer (no react-query)
- All DB access goes through `src/services/*.service.ts`, which share one Supabase client (`src/services/supabase.ts` → `src/lib/client.ts`). The client uses **only the publishable key** (`VITE_SUPABASE_*` in `.env.local`); it ships to the browser and is safe **because security is enforced by Postgres RLS**, not the key. Never introduce a `service_role` key here.
- **Service module pattern** (follow it for any new entity — see `leads.service.ts`):
  - Export the row `type` and a `New<Entity>` input type; export both from `src/services/index.ts` (values and `type`s) so consumers import from `@/services`, not the file directly.
  - Define a module-level `<ENTITY>_COLUMNS` select string that includes embedded joins (e.g. `project:projects(project_name, currency)`); reuse it across `list`/`create`/`update` so returned rows stay uniformly shaped.
  - Export a single `<entity>Service` object of `async` methods (`list`, `create`, `update…`). Each awaits `supabase.from(...)`, does `if (error) throw error`, and returns typed data (`as <Type>`). Let errors propagate — callers handle them with a Sonner toast. Also provide `export default`.
- Pages fetch on mount with a `useEffect` + local `active` flag guard, set `useState`, and surface errors via **Sonner** `toast.error`. Mutations optimistically update local state and revert on failure (see `src/components/pipeline/pipeline-board.tsx`). Follow this pattern rather than adding a data-fetching library.

### State (Zustand)
- Stores in `src/stores/`. `auth-store.ts` holds the signed-in profile (`useAuthStore((s) => s.profile)`, e.g. `s.profile?.role === "admin"`). Per-page table view state (sorting, filters, column sizing) is persisted in dedicated stores like `leads-store.ts`.

### Layout patterns
- **Header slot**: pages inject header actions by rendering `<HeaderSlot>…</HeaderSlot>` (`src/components/layout/header-slot.tsx`); it portals into a target `MainLayout` renders in the header, so React context still works.
- **Page title** derives from the URL via `usePageTitle()`, overridable through `page-title-store.ts`.

### Styling
- **Tailwind CSS v4 with no `tailwind.config`** — theme is configured inline in `src/index.css` using `@theme` and OKLCH design tokens (light + dark; brand palette is teal-green). Adjust tokens there.
- `src/index.css` imports `shadcn/tailwind.css`, which provides extra utilities like `scroll-fade` (masked scroll edges) beyond the base Tailwind set.
- UI primitives are shadcn/ui in `src/components/ui/`; compose class names with `cn()` from `src/lib/utils.ts`. Match the existing component idiom (CVA variants, `data-slot` attributes).

### Domain: leads & pipeline
- The lead lifecycle is an 11-value `lead_stage` enum. Central metadata — labels, the 4 pipeline groupings, and display order — lives in `src/lib/lead-format.ts` (`LEAD_STAGE_LABELS`, `LEAD_STAGE_GROUPS`, `LEAD_STAGE_ORDER`) alongside budget formatters. Reuse these; the Leads table and Pipeline Kanban both build off them.
- `lead_stage` is also mutated server-side: DB triggers sync it from appointment activity (`supabase/migrations/…_sync_lead_stage_from_appointments.sql`), so stage can change without a client write.
