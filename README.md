# Megaterra Client

Web client for **Megaterra**, a real‑estate CRM: it manages leads, a sales pipeline, appointments, projects, inventory, quotes, and closing formalities for a real‑estate sales team.

Single‑page React app built with Vite, TanStack Router, Tailwind CSS v4, and shadcn/ui, backed by Supabase (Postgres + Auth) with row‑level security.

## Tech stack

| Area | Choice |
| ---- | ------ |
| Framework | React 19 + TypeScript |
| Build tool | Vite 8 |
| Routing | TanStack Router (file‑based, auto‑generated route tree) |
| Styling | Tailwind CSS v4, shadcn/ui + Radix/Base UI, `tw-animate-css` |
| State | Zustand |
| Data / backend | Supabase (`@supabase/ssr`, `@supabase/supabase-js`) |
| Tables / charts | TanStack Table, Recharts |
| Drag & drop | `@dnd-kit/core` (pipeline board) |
| Other | date-fns, jsPDF (quotes), Sonner (toasts), Lucide icons |

## Features

Each page lives under `src/routes/_authenticated/` and is gated behind Supabase auth:

- **Dashboard** (`/home`) — overview.
- **Leads** (`/leads`) — sortable/filterable data table with pipeline‑group summary cards.
- **Pipeline** (`/pipeline`) — Kanban board with one column per lead stage; drag a card to change its stage (persisted to Supabase).
- **Appointments** (`/appointments`) — scheduling; syncs lead stages via DB triggers.
- **Projects** (`/projects`, `/projects/:projectId`) — real‑estate developments.
- **Inventory** (`/inventory`) — units per project.
- **Quotes** (`/quotes`) — quote builder with PDF export.
- **Clients** (`/clients`) & **Formalities** (`/formalities`) — closing pipeline.
- **Settings** (`/settings`) & **Account** (`/account`).

## Prerequisites

- **Node.js 24+** (developed on v24.11).

## Getting started

```bash
# 1. Install dependencies
pnpm install

# 2. Configure environment (see below)
#    create .env.local with your Supabase URL + publishable key

# 3. Start the dev server (http://localhost:5173)
pnpm dev
```

### Environment variables

Create `megaterra-client/.env.local` (gitignored) with:

```bash
VITE_SUPABASE_URL=https://<your-project-ref>.supabase.co
VITE_SUPABASE_PUBLISHABLE_KEY=sb_pub_xxxxxxxxxxxxxxxxxxxx
```

Both are consumed in [`src/lib/client.ts`](src/lib/client.ts). The publishable key is safe to ship to the browser — **all data access is protected by Postgres row‑level security**, not by hiding the key. Never put a `service_role` key in a `VITE_` variable.

```
│   ├── __root.tsx
│   ├── _authenticated.tsx      # Auth guard + MainLayout wrapper
│   └── _authenticated/<page>/  # One folder per page
├── components/
│   ├── ui/            # shadcn/ui primitives (Card, Button, ScrollArea, …)
│   ├── layout/        # MainLayout, header slot, current-user
│   ├── pipeline/      # Kanban board, column, card
│   └── …              # Feature components (leads, projects, quoter, …)
├── services/          # Supabase data layer (leads, projects, appointments, …)
├── stores/            # Zustand stores (auth, sidebar, per-page table state)
├── hooks/             # Shared hooks (usePageTitle, …)
├── lib/               # Supabase client, formatters, utils
├── assets/            # Images, topography background
├── routeTree.gen.ts   # Auto-generated by the router plugin — do not edit

```

## Architecture notes

- **Routing** is file‑based: add a folder under `src/routes/_authenticated/` and export a `Route` via `createFileRoute`. `routeTree.gen.ts` regenerates automatically — don't edit it. Add the matching sidebar entry in [`src/components/app-sidebar.tsx`](src/components/app-sidebar.tsx).
- **Auth** is enforced in [`_authenticated.tsx`](src/routes/_authenticated.tsx): unauthenticated users are redirected to `/login`, and the signed‑in profile is hydrated into the Zustand auth store.
- **Data access** goes through the `*.service.ts` modules, which call a single shared Supabase client. Pages load on mount with a `useEffect` + `active`‑flag pattern and surface errors via Sonner toasts.
- **Theming** uses Tailwind v4 with OKLCH design tokens defined in `src/index.css` (light + dark); the brand palette is teal‑green.

## Database

Schema, migrations, RLS policies, and the first‑admin bootstrap are documented in **[`supabase/README.md`](supabase/README.md)**. Migrations are applied with the Supabase CLI (`npx supabase db push`).

## Building for production

```bash
pnpm build     # outputs to dist/
pnpm preview   # serve the built app locally to verify
```

`dist/` is a static bundle deployable to any static host (Vercel, Netlify, Cloudflare Pages, S3, …). Set the two `VITE_` env vars in your host's build environment.
